# 第08章 高级技巧

## 本章导读
当基础断言、参数化测试和 Mockito 已经能够覆盖大部分日常场景后，仍然会遇到一些更复杂的问题。例如：一个测试需要同时验证多个结果；某个测试只应在特定条件下运行；某段逻辑需要重复执行以确认稳定性；又或者测试数据需要在运行时动态生成。这些需求并不高频，但在合适场景下，它们能显著提升测试表达力与覆盖能力。本章介绍 JUnit 5 中几种常见高级技巧，并重点讨论它们的适用边界。

## 学习目标
- 理解 `assertAll`、`assertDoesNotThrow`、`assumeTrue` 的使用场景。
- 掌握 `@RepeatedTest` 与 `@TestFactory` 的基本写法。
- 能区分“增强表达能力”和“增加测试复杂度”的边界。
- 结合现有示例理解这些能力在工程中的实际价值。

## 8.1 为什么需要高级技巧
高级技巧的价值不在于“功能更多”，而在于解决基础写法难以优雅表达的问题。  
典型情况包括：
- 一个业务动作需要同时验证多个结果。
- 某段逻辑在特定环境下才有意义。
- 某个测试需要多次执行以验证稳定性。
- 测试集合需要根据运行时数据动态生成。

如果仍然强行使用最基础写法，结果通常会出现：
- 测试结构臃肿
- 重复代码增加
- 报告信息分散
- 可读性下降

## 8.2 `assertAll`：聚合多个断言
### 8.2.1 解决什么问题
在一个业务动作中，有时需要同时验证多个相关结果。  
如果逐个断言，前一个失败后后续断言就不会继续执行，结果只能看到“第一个问题”，不利于一次性定位整体失败信息。

`assertAll` 的作用是：
- 聚合多个断言
- 即使其中一个失败，也继续执行其他断言
- 最后统一汇总失败信息

### 8.2.2 示例
对应测试类：
- `src/test/java/com/company/training/core/AssertionsAndAssumptionsTest.java`

示例中使用：

```java
assertAll("核心结果检查",
        () -> assertEquals(9, calculator.add(4, 5)),
        () -> assertEquals("Alice", normalizer.normalizeName("aLiCe")),
        () -> assertEquals(80.0, discountCalculator.calculateFinalPrice(100, "VIP"), 0.0001)
);
```

这段代码表达的是：  
在一个业务检查点中，同时验证多个核心结果。

### 8.2.3 什么时候适合使用
- 多个断言共同构成一次完整业务验证。
- 希望一次性看到所有失败项。
- 多个断言属于同一业务上下文，而非彼此独立的不同场景。

### 8.2.4 什么时候不适合使用
- 多个断言实际上对应不同测试意图。
- 为了“省测试方法数量”而把不相关断言硬凑在一起。

## 8.3 `assertDoesNotThrow`：显式声明“不应抛错”
### 8.3.1 作用
有些业务方法的核心要求并不是“返回某个具体值”，而是“在合法输入下必须稳定执行，不允许抛异常”。  
这类场景中，`assertDoesNotThrow` 可以更准确表达测试意图。

### 8.3.2 示例
在 `AssertionsAndAssumptionsTest` 中：

```java
assertDoesNotThrow(() -> discountCalculator.calculateFinalPrice(88.0, "NORMAL"));
```

这不是在验证结果值，而是在验证：  
对于合法输入，该方法至少应当稳定执行。

### 8.3.3 使用建议
- 当“不抛异常”本身就是业务要求时使用。
- 若业务还要求精确结果，应在此基础上补充结果断言，而不是只停留在“不报错”。

## 8.4 `assumeTrue`：前置条件不满足时跳过
### 8.4.1 作用
有些测试只在特定前提下才有意义，例如：
- 特定系统属性开启时
- 某类外部依赖可用时
- 某些环境变量存在时

这类场景如果直接失败，未必表示代码错误，而是测试执行条件不满足。  
`assumeTrue` 的作用就是在条件不满足时跳过测试，而不是判定失败。

### 8.4.2 示例
在 `AssertionsAndAssumptionsTest` 中：

```java
String runIntegrationLikeCase = System.getProperty("run.integration.like.case", "false");
assumeTrue("true".equalsIgnoreCase(runIntegrationLikeCase),
        "仅用于演示 assumeTrue：默认会被跳过");
```

如果条件不满足，该测试会被标记为跳过。

### 8.4.3 使用建议
- 用于“执行前提不满足”的情况。
- 不应用它规避真实失败问题。
- 如果测试经常依赖 `assumeTrue` 才能跑，通常说明测试边界设计还不够理想。

## 8.5 `@RepeatedTest`：重复执行验证稳定性
### 8.5.1 作用
某些逻辑并不复杂，但希望通过重复执行确认其稳定性。  
例如：
- 依赖瞬时状态的轻量逻辑
- 希望验证重复执行时上下文是否正确
- 演示 JUnit 提供的重复执行能力

### 8.5.2 示例
对应测试类：
- `src/test/java/com/company/training/core/RepeatedAndDynamicTest.java`

示例中：

```java
@RepeatedTest(value = 3, name = "第 {currentRepetition} / {totalRepetitions} 次")
void shouldRunRepeatedly(RepetitionInfo repetitionInfo) {
    assertTrue(repetitionInfo.getCurrentRepetition() >= 1);
    assertTrue(repetitionInfo.getCurrentRepetition() <= repetitionInfo.getTotalRepetitions());
}
```

该测试展示了：
- 同一测试重复运行多次
- 可通过 `RepetitionInfo` 获取当前轮次信息

### 8.5.3 使用建议
- 用于验证重复执行是否稳定。
- 不要把它当作随机“碰碰运气”的手段。
- 如果一个测试必须重复很多次才放心，往往说明逻辑中存在不稳定因素，应优先治理根因。

## 8.6 `@TestFactory`：动态生成测试
### 8.6.1 作用
有些测试数据在编码时无法自然写成固定注解参数，或者更适合在运行时通过代码生成。  
此时可使用 `@TestFactory` 返回一组 `DynamicTest`。

### 8.6.2 示例
在 `RepeatedAndDynamicTest` 中：

```java
List<String> rawNames = Arrays.asList(" TOM", "jErrY", "  spike ");
List<String> expectedNames = Arrays.asList("Tom", "Jerry", "Spike");

return Arrays.asList(0, 1, 2).stream()
        .map(i -> DynamicTest.dynamicTest(
                "动态用例-" + i,
                () -> assertEquals(expectedNames.get(i), normalizer.normalizeName(rawNames.get(i)))
        ))
        .collect(Collectors.toList());
```

这个例子说明：
- 测试可以在运行时被构造出来
- 每个动态用例都拥有独立名称
- 每个动态用例都可以包含独立断言逻辑

### 8.6.3 适用场景
- 测试数据集本身由代码生成。
- 测试数量在运行前不固定。
- 需要用更灵活方式组织测试集合。

### 8.6.4 不适用场景
- 只是为了炫技而替代普通参数化测试。
- 固定数据完全可以用 `@ParameterizedTest` 更清晰表达时。

在大多数业务项目中，优先顺序通常是：
1. 普通 `@Test`
2. `@ParameterizedTest`
3. `@TestFactory`

## 8.7 两个示例类分别解决什么问题
### 8.7.1 `AssertionsAndAssumptionsTest`
对应文件：
- `src/test/java/com/company/training/core/AssertionsAndAssumptionsTest.java`

该类重点演示：
- `assertAll`：聚合多个结果断言
- `assumeTrue`：条件不满足时跳过
- `assertDoesNotThrow`：声明合法输入不应抛异常

### 8.7.2 `RepeatedAndDynamicTest`
对应文件：
- `src/test/java/com/company/training/core/RepeatedAndDynamicTest.java`

该类重点演示：
- `@RepeatedTest`：重复执行
- `@TestFactory`：动态生成测试

两者结合后，覆盖了本章的主要高级能力。

## 8.8 使用高级技巧时的常见误区
### 8.8.1 为了“看起来高级”而使用高级特性
如果基础 `@Test` 就足够清晰，强行使用 `@TestFactory` 或复杂断言只会提高维护成本。

### 8.8.2 用 `assertAll` 聚合不相关断言
这会让一个测试承担多个测试意图，降低失败定位效率。

### 8.8.3 用 `assumeTrue` 掩盖真实问题
跳过测试并不等于修复问题。  
如果一个测试本应失败，却通过 assumption 被跳过，那么问题仍然存在。

### 8.8.4 把重复测试当作稳定性补丁
如果某段逻辑本身不稳定，重复执行只是在放大不稳定，而不是解决问题。

## 8.9 实践建议
- 优先使用最简单且能清晰表达意图的测试方式。
- `assertAll` 适合一个业务动作下的多结果检查。
- `assumeTrue` 只用于执行前提控制，不用于规避失败。
- `@RepeatedTest` 用于验证稳定性，不用于掩盖问题。
- `@TestFactory` 只在动态构造测试确有价值时使用。

## 本章小结
高级技巧的价值，在于提升测试表达能力和覆盖灵活性，而不是增加测试“技术含量”。真正高质量的测试，关键仍然是：测试意图明确、结构清晰、行为可验证。只有在基础写法已经不足以准确表达业务需求时，这些高级能力才值得引入。

## 思考与练习
- 用 `assertAll` 改造一个现有多断言测试，并判断这些断言是否真的属于同一业务上下文。
- 为动态测试增加一组边界输入，并观察测试报告名称是否足够清晰。
- 思考：某个测试如果必须依赖 `assumeTrue` 才能执行，是否说明它更适合作为集成测试而不是单元测试？
