# 第06章 参数化测试进阶

## 本章导读
基础参数化测试解决的是“相同规则、多组简单输入”的问题，但在真实业务中，测试数据往往不止一个字段，也不总能用几行字符串简单表达。例如折扣计算、状态转换、权限组合、阈值判断等场景，通常需要同时传入多个参数，甚至需要先对数据做分组建模。此时，`@MethodSource` 就成为参数化测试从“简单可用”走向“工程可维护”的关键能力。

## 学习目标
- 理解 `@MethodSource` 的适用场景与核心价值。
- 掌握 `Stream<Arguments>` 的基本组织方式。
- 能将复杂业务规则抽象为结构化测试数据。
- 理解如何按“正常/异常/边界”组织复杂参数集合。

## 6.1 为什么需要进阶参数化测试
在基础参数化测试中，`@CsvSource`、`@ValueSource` 和 `@NullSource` 已经足以覆盖大量简单场景。但当测试复杂度提升时，会出现以下问题：
- 测试参数变多，`@CsvSource` 可读性下降。
- 测试数据需要分组或复用。
- 参数中包含对象、枚举或组合条件。
- 测试数据本身具有业务语义，单纯写成一行字符串不利于维护。

这说明，参数化测试不仅是“换一种写法”，而是需要建立测试数据建模能力。

## 6.2 `@MethodSource` 的作用
`@MethodSource` 允许通过一个 Java 方法返回测试数据集合，再由 JUnit 将这些数据逐组注入测试方法。

最常见的返回形式是：

```java
static Stream<Arguments> cases() {
    return Stream.of(
        arguments(100.0, "VIP", 80.0),
        arguments(100.0, "NORMAL", 100.0)
    );
}
```

与基础参数源相比，`@MethodSource` 的优势在于：
- 可读性更强
- 扩展性更好
- 更适合多参数与复杂参数
- 可以把测试数据本身当作“业务规则模型”来维护

## 6.3 什么时候优先选择 `@MethodSource`
以下场景优先考虑 `@MethodSource`：
- 一个测试需要传入多个参数。
- 输入值较长或包含空格、特殊字符，`@CsvSource` 不直观。
- 测试数据需要分成多类组织。
- 需要根据 Java 代码动态生成测试数据。
- 未来可能会持续扩展新的规则组合。

相反，如果只是一个或两个简单值的快速批量验证，仍然建议优先使用基础参数源，因为那样更直接。

## 6.4 `Arguments` 的建模思路
编写 `@MethodSource` 时，最重要的不是语法，而是如何理解“什么是一组业务事实”。

以折扣规则为例，一条测试数据至少包含三部分：
- 原始金额
- 用户等级
- 期望价格

这三项放在一起，才构成一个完整的断言单元。  
如果把它们拆散，就会破坏业务语义。

因此，一个好的 `Arguments` 设计，应满足：
- 每组参数完整表达一个业务场景
- 参数顺序稳定、含义清晰
- 后续新增规则时容易追加

## 6.5 案例解析：`DiscountCalculatorMethodSourceTest`
对应业务类：
- `src/main/java/com/company/training/core/DiscountCalculator.java`

对应测试类：
- `src/test/java/com/company/training/core/DiscountCalculatorMethodSourceTest.java`

该示例是 `@MethodSource` 的典型业务化应用。它将折扣计算规则拆分为两类测试：
- 合法输入：验证不同等级下价格计算结果是否正确
- 非法输入：验证空等级、未知等级是否触发异常

这种拆分方式具有很强的工程价值，因为它让“规则正确性”和“输入合法性”成为两个独立关注点。

## 6.6 合法输入场景的组织方式
在合法输入场景中，一组测试数据通常表示：
- 输入金额
- 会员等级
- 最终价格

例如：
- `100.0 + VIP -> 80.0`
- `100.0 + INTERN -> 90.0`
- `100.0 + NORMAL -> 100.0`

这些场景共享同一条测试逻辑，因此非常适合放入同一个 `Stream<Arguments>` 中。

这样做的优点是：
- 折扣规则集中展示
- 新增等级时只需增加一行数据
- 断言逻辑保持不变

## 6.7 异常场景的组织方式
对于非法输入，通常不再关心结果值，而关心是否触发正确异常。  
此时可将测试拆成另一个参数化方法，并单独提供一组非法输入数据。

这样比把合法/非法混在一个测试方法里更清晰，原因是：
- 断言逻辑不同
- 场景目标不同
- 失败后定位更明确

这体现了一个重要原则：  
参数化测试是在复用“同一条测试规则”，而不是强行把所有数据装进一个方法。

## 6.8 `@MethodSource` 与 `@CsvSource` 的区别
两者都能提供多组数据，但关注点不同：

- `@CsvSource`
  - 更适合简单、短小、静态的文本型数据
  - 上手快，适合基础场景
- `@MethodSource`
  - 更适合复杂、结构化、可扩展的数据
  - 适合真实业务规则测试

可以把它们理解为：
- `@CsvSource` 偏“轻量录入”
- `@MethodSource` 偏“结构化建模”

## 6.9 常见问题
### 6.9.1 一个 `MethodSource` 承担过多职责
如果一个数据源里同时混合了正常值、异常值、边界值，而且测试逻辑也不同，那么数据源虽然复用了，但可读性会明显下降。

### 6.9.2 `Arguments` 顺序不清晰
参数一多时，如果没有明确顺序约定，阅读和维护都会变得困难。  
建议保持参数顺序与业务表达顺序一致。

### 6.9.3 测试方法名和数据源名缺乏业务语义
像 `data1()`、`cases()` 这样的命名虽然可运行，但可读性有限。  
如果数据源本身能表达业务含义，后续维护会轻松很多。

## 6.10 实践建议
- 多参数规则优先考虑 `@MethodSource`。
- 合法数据和非法数据建议分成不同方法组织。
- 测试数据应围绕业务语义组织，而不是围绕技术便利组织。
- 当测试数据开始具有明显结构时，应把它当作“规则清单”维护。

## 本章小结
`@MethodSource` 让参数化测试从简单值验证提升到了业务规则建模层面。它特别适合多参数、复杂规则、长期维护的测试场景。掌握 `@MethodSource` 后，参数化测试就不再只是减少重复代码的手段，而成为表达业务规则的重要方式。
