# 第01章 培训目标

## 本章导读
单元测试常被误解为“额外工作”或“为了覆盖率而写的代码”。这种理解过于狭窄。对于持续演进的 Java 项目而言，单元测试首先是一种工程能力，其核心价值在于让代码变更具备可验证性，让重构具备安全边界，让团队协作建立在可执行事实之上。本章作为整套培训材料的起点，重点说明本次培训为什么要做、要解决什么问题，以及完成培训后应具备哪些能力。

## 学习目标
- 理解单元测试在质量保障、开发效率和团队协作中的作用。
- 明确本次培训覆盖的技术边界与实践范围。
- 建立“先设计可测试代码，再编写稳定测试”的基本意识。
- 形成后续章节的学习坐标，知道每一章要解决什么问题。

## 1.1 为什么要做单元测试
在业务迭代频繁、需求变化持续发生的环境中，代码的主要风险并不是“第一次写错”，而是“后续改动破坏已有行为”。单元测试的价值，正是在这种持续变更的场景中体现出来。

单元测试主要提供以下三类价值：
- 质量保障：将问题暴露在开发阶段，而不是测试环境或生产环境。
- 研发效率：为重构和优化提供可重复验证机制，减少人工回归成本。
- 协作标准：通过测试用例把业务规则显式化，降低沟通歧义。

从工程视角看，单元测试并不是独立存在的。它通常位于如下链路中：

```text
需求理解 -> 编码实现 -> 单元测试 -> 本地验证 -> CI 执行 -> 代码评审 -> 发布上线
```

如果缺少单元测试，代码评审往往只能停留在“看起来没问题”的层面；如果具备高质量测试，评审就可以围绕行为正确性、边界覆盖和设计质量展开。

## 1.2 本次培训要解决的问题
本次培训不是简单介绍几个注解，而是围绕以下几个实际问题展开：
- 一个业务方法该如何设计测试场景。
- 如何验证正常路径、异常路径和边界路径。
- 如何隔离数据库、邮件、HTTP 调用等外部依赖。
- 在 Spring Boot 和 MyBatis 项目中，如何选择合适的测试层级。
- 如何通过测试报告和覆盖率报告持续治理测试质量。

换句话说，本次培训的重点不是“会写一个测试”，而是“会持续写对测试”。

## 1.3 适用对象
本套材料适用于以下几类开发场景：
- 需要在 Java 后端项目中补充基础单元测试的开发人员。
- 正在使用 Maven 管理项目、希望规范测试结构的团队。
- 使用 Spring Boot、MyBatis 等常见框架，并希望建立测试分层认知的工程团队。
- 希望将单元测试纳入日常 Code Review 和 CI 流程的项目负责人或骨干开发人员。

即使当前项目历史包袱较重、测试覆盖率较低，本套材料也仍然适合作为起步模板，因为示例覆盖了“纯业务逻辑”“依赖隔离”“Web 层切片”“数据库访问”四个高频场景。

## 1.4 本次培训范围
本项目以 Maven + JDK 1.8 为基础，围绕实际工程中最常见的测试需求展开，覆盖如下内容：
- JUnit 5 的基础写法与常用断言。
- 参数化测试、生命周期管理和动态测试等进阶能力。
- Mockito 的依赖隔离、交互验证、参数捕获与顺序校验。
- Spring Boot 的 Service 单测、Controller 切片测试与上下文测试。
- MyBatis 的 Mapper CRUD 测试与 Service 集成测试。
- 测试报告与覆盖率报告生成。
- 测试最佳实践、反模式与落地建议。

本次培训不重点展开以下内容：
- UI 自动化测试。
- 性能压测与压力测试。
- 分布式集成链路测试。
- 复杂测试平台建设。

这样限定范围的目的，是将注意力集中在最常见、最容易立即落地的单元测试实践上。

## 1.5 预期产出
完成全部章节后，应具备以下能力：
- 能为核心业务类编写结构清晰、断言准确的单元测试。
- 能根据场景选择纯单测、切片测试或集成测试。
- 能识别不可测试代码的常见设计问题，并给出改进方向。
- 能在 Spring Boot 与 MyBatis 场景下编写可运行的测试示例。
- 能生成并阅读测试报告与覆盖率报告。
- 能在团队工程中推动测试规范逐步落地。

如果把结果再进一步拆解，可以将培训目标理解为三层：
- 会写：掌握语法、框架和案例。
- 会用：能迁移到实际业务代码。
- 会判断：知道什么该测、什么不该测、什么需要优先测。

## 1.6 建议学习路径
为了获得更好的理解效果，建议按照以下顺序阅读和实践：
1. 先掌握 JUnit 基础写法与断言模型。
2. 再学习参数化测试、Mock 和高级技巧。
3. 然后进入 Spring Boot 与 MyBatis 的框架化测试场景。
4. 最后回到最佳实践、报告治理与团队落地。

这样的顺序符合从“语言与工具”到“工程与治理”的学习路径，可以避免在尚未掌握基础时直接进入框架复杂度过高的章节。

## 1.7 如何使用这套材料
这套材料既可以用于集中培训，也可以用于自学或团队内部分享。推荐使用方式如下：
- 先阅读对应章节文档，理解概念与目标。
- 再对照项目中的测试类运行示例。
- 阅读测试代码与被测代码之间的对应关系。
- 修改部分示例输入或逻辑，观察测试结果变化。
- 最后将同样的方法迁移到真实业务代码。

如果只是阅读文档而不运行测试，理解会停留在概念层；只有在代码和测试之间来回切换，才能真正建立测试思维。

## 本章小结
单元测试的本质不是“补代码数量”，而是“建立变更可验证机制”。本章明确了本次培训的价值、范围、对象和目标。后续章节将从最基础的 JUnit 写法开始，逐步进入依赖隔离、框架测试、数据库测试以及测试治理实践。

## 思考与练习
- 当前项目中，哪些模块最需要优先补充单元测试？原因是什么？
- 如果只能先覆盖 20% 的代码，应如何确定优先级？
- 在现有团队协作模式下，单元测试最可能卡在哪个环节：认知、时间、工具还是规范？
