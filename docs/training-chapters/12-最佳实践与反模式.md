# 第12章 最佳实践与反模式

## 本章导读
当测试体系搭建到一定程度后，真正拉开团队差距的往往不是“是否写了测试”，而是“测试是否具备长期维护价值”。高质量测试可以持续提供反馈、支撑重构、降低回归成本；低质量测试则会变成负担，既拖慢开发，又不能提供可信保护。本章聚焦测试实践中的经验总结，分别从最佳实践、反模式、代码可测试性和评审视角四个方面，建立一套可执行的质量判断标准。

## 学习目标
- 建立可执行的测试质量判断框架。
- 识别常见反模式，并理解其风险。
- 理解“可测试性”为什么是设计问题，而不仅是测试问题。
- 形成可用于代码评审的检查视角。

## 12.1 什么是高质量测试
高质量测试通常具备以下特征：
- 可读：读者能快速理解它在验证什么。
- 可信：测试通过意味着行为真的符合预期。
- 稳定：重复执行结果一致，不依赖偶然条件。
- 可维护：业务变化后，修改成本合理。

这四点缺一不可。  
例如，一个覆盖率很高但极难理解的测试集合，长期来看依然会拖垮维护效率。

## 12.2 最佳实践
### 12.2.1 一个测试只验证一个明确行为
测试粒度过大时，失败定位会变慢。  
让一个测试只承担一个核心判断，有助于快速理解失败原因。

### 12.2.2 测试命名体现业务语义
测试名称应该回答“在什么条件下，期望发生什么结果”，而不是只描述技术动作。

### 12.2.3 测试数据最小化且可读
测试数据的目标不是“越真实越好”，而是“足够表达业务场景且容易理解”。

### 12.2.4 断言聚焦业务结果
应优先断言业务意义明确的结果和副作用，而不是紧盯实现细节。

### 12.2.5 测试必须稳定可重复
如果一个测试偶尔通过、偶尔失败，它的价值会迅速下降。  
稳定性是测试能否纳入日常工程流的前提。

## 12.3 常见反模式
### 12.3.1 在测试中复制业务实现逻辑
如果测试内部再次写了一遍复杂计算、判断或循环，相当于把 bug 复制了两份。  
这类测试往往“看起来很严谨”，但可信度反而低。

### 12.3.2 过度 Mock
把所有对象都 mock 掉，会让测试失去真实业务约束，只剩下人为拼接的流程脚本。

### 12.3.3 依赖不稳定因素
典型包括：
- 系统时间
- 随机数
- 网络状态
- 外部数据库历史数据

一旦测试依赖这些因素，就容易出现不稳定问题。

### 12.3.4 忽视失败测试
长期存在失败或跳过测试，会直接损害测试体系的可信度。  
一旦团队习惯忽略红灯，测试就会逐渐失去工程意义。

## 12.4 如何保证代码的可测试性
可测试性不是测试阶段才考虑的问题，而是编码阶段的设计结果。  
测试写得轻松，往往说明业务代码结构合理；测试写得异常艰难，通常也暴露出设计问题。

### 12.4.1 依赖通过构造器注入
将仓储、消息、HTTP 客户端等依赖通过构造器传入，而不是在方法内部直接创建。  
这样在测试中可以自然替换为 mock。

示例（当前项目）：
- `src/main/java/com/company/training/service/UserService.java`
- `src/test/java/com/company/training/service/UserServiceMockitoTest.java`

`UserService` 通过构造器接收 `UserRepository` 和 `EmailSender`，因此可以直接用 Mockito 隔离依赖。

### 12.4.2 将业务规则与框架代码分离
越“纯”的业务逻辑，越容易被快速测试。  
例如：
- `src/main/java/com/company/training/core/DiscountCalculator.java`
- `src/test/java/com/company/training/core/DiscountCalculatorMethodSourceTest.java`

该类不依赖 Spring 容器，因此测试成本极低。

### 12.4.3 显式处理边界与异常
如果方法对非法输入没有明确策略，测试就难以建立稳定预期。  
清晰的异常或默认值策略，本身也是提升可测试性的设计动作。

示例：
- `Calculator.divide` 对除数为 0 抛出异常
- 测试类：`src/test/java/com/company/training/core/CalculatorTest.java`

### 12.4.4 将不稳定因素隔离在边界层
时间、随机数、网络、数据库不应无约束地散落在核心逻辑中。  
更合理的做法是通过接口抽象这些依赖，使核心逻辑只消费确定性输入。

示意：

```java
interface ClockProvider {
    long now();
}

class OrderService {
    private final ClockProvider clockProvider;

    OrderService(ClockProvider clockProvider) {
        this.clockProvider = clockProvider;
    }
}
```

这样测试时可以传入固定时间实现，避免因时间变化导致不稳定。

## 12.5 如何从评审视角审查测试质量
在代码评审中，测试代码也应被认真审查，而不是“只要有就行”。  
可以重点关注以下问题：
- 是否覆盖了正常路径、异常路径和边界路径。
- 断言是否真正表达业务规则。
- 是否过度依赖 Mock 或实现细节。
- 测试是否稳定、可重复。
- 测试结构是否清晰，命名是否可读。

如果测试只能证明“代码跑了一遍”，而不能证明“行为正确”，那它的价值就非常有限。

## 12.6 与当前项目示例的对应关系
当前项目已经展示了若干良好实践：
- 纯逻辑类测试：
  - `CalculatorTest`
  - `StringNormalizerParameterizedTest`
  - `DiscountCalculatorMethodSourceTest`
- 服务层依赖隔离测试：
  - `UserServiceMockitoTest`
  - `UserServiceArgumentCaptorTest`
  - `UserServiceInteractionOrderTest`
- Spring / MyBatis 分层测试：
  - `GreetingControllerWebMvcTest`
  - `EmployeeMapperMybatisTest`
  - `EmployeeServiceSpringBootTest`

这些示例共同说明：  
好的测试不是统一一种写法，而是根据目标选择合理策略。

## 12.7 实践建议
- 优先为高风险、核心路径和频繁变更模块补测试。
- 在编写业务代码时同步考虑可测试性，而不是事后补救。
- 对测试失败保持高敏感度，避免长期带病运行。
- 将测试质量纳入代码评审，不把测试视为附属物。

## 本章小结
高质量测试应同时满足可信、可读、稳定、可维护四个条件。最佳实践强调聚焦、清晰和稳定，反模式则通常表现为过度复杂、过度依赖环境或过度绑定实现。更进一步地，测试质量并不只取决于测试代码本身，还取决于业务代码是否具备良好的可测试性设计。
