# 第12章 最佳实践与反模式

## 本章导读
测试质量决定测试价值。本章总结可复用实践与常见误区，帮助形成统一质量标准。

## 学习目标
- 建立可执行的测试质量检查框架。
- 识别并规避高频反模式。

## 12.1 最佳实践
- 一个测试只验证一个业务行为。
- 测试命名体现业务语义。
- 测试数据最小化并具可读性。
- 断言关注业务结果而非实现细节。
- 测试必须稳定、可重复。

## 12.2 常见反模式
- 在测试中复制业务实现逻辑。
- 过度 Mock 导致测试失真。
- 依赖外部网络、系统时间、随机值导致不稳定。
- 忽视失败测试，长期处于“红灯”状态。

## 12.3 如何保证代码的可测试性
可测试性不是测试阶段才考虑的问题，而是编码阶段的设计结果。以下原则可以显著降低测试成本。

### 12.3.1 依赖通过构造器注入
将外部依赖（仓储、消息、HTTP 客户端）通过构造器注入，而不是在方法内直接 `new`。  
这样测试中可以直接替换为 mock。

示例（当前项目）：
- 可测试写法：`src/main/java/com/company/training/service/UserService.java`
  - `UserRepository` 与 `EmailSender` 均由构造器注入。
- 对应测试：`src/test/java/com/company/training/service/UserServiceMockitoTest.java`
  - 可直接使用 `@Mock` + `@InjectMocks` 验证业务逻辑。

### 12.3.2 业务逻辑与框架细节分离
将规则判断放在普通 Java 类中，避免与框架 API 强耦合。  
规则越“纯”，测试越简单、执行越快。

示例（当前项目）：
- `src/main/java/com/company/training/core/DiscountCalculator.java`
- `src/test/java/com/company/training/core/DiscountCalculatorMethodSourceTest.java`

该类不依赖 Spring 上下文，单测可毫秒级完成。

### 12.3.3 显式处理边界与异常分支
方法应对无效输入有明确行为（抛异常或返回默认值），便于测试建立稳定预期。

示例（当前项目）：
- `Calculator.divide` 对除数为 0 抛异常。  
- 测试：`src/test/java/com/company/training/core/CalculatorTest.java`

### 12.3.4 将不稳定因素隔离在边界层
时间、随机数、网络、数据库属于不稳定因素。  
建议把它们放在边界层，通过接口抽象隔离，在核心逻辑中只处理确定性输入。

示例（模式示意）：
```java
interface ClockProvider {
    long now();
}

class OrderService {
    private final ClockProvider clockProvider;
    OrderService(ClockProvider clockProvider) { this.clockProvider = clockProvider; }
}
```

测试时可注入固定时间实现，避免“今天通过、明天失败”。

## 12.4 质量检查建议
在评审测试代码时可检查：
- 场景覆盖是否包含正常/异常/边界。
- 断言是否精准且可解释。
- 是否存在不必要复杂度。
