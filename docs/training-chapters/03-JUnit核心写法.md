# 第03章 JUnit 核心写法

## 本章导读
单元测试真正考验的不是“是否知道注解名字”，而是能否把业务规则准确地翻译成测试代码。写得好的测试，应当像一份简洁明确的行为说明书：给定什么输入，执行什么动作，预期得到什么结果。本章从最基础的 JUnit 写法开始，建立一套可重复、可扩展的测试表达模型。

## 学习目标
- 掌握 AAA（Arrange-Act-Assert）结构，并能据此组织测试代码。
- 能区分正常路径、异常路径和边界路径。
- 理解常用断言的适用场景与表达意图。
- 学会用清晰命名和简洁结构提升测试可读性。

## 3.1 单元测试的基本构成
一个完整的单元测试通常包含三个部分：
- 输入条件
- 被测行为
- 结果验证

从代码结构上看，最常见且最稳定的组织方式就是 AAA 模型。

## 3.2 AAA 写作模型
AAA 是单元测试中最常见的组织方式：
- Arrange：准备输入、依赖和前置条件。
- Act：执行被测方法。
- Assert：验证结果与行为。

例如下面这种结构，就具有非常好的可读性：

```java
// Arrange
Calculator calculator = new Calculator();

// Act
int result = calculator.add(3, 4);

// Assert
assertEquals(7, result);
```

AAA 的价值在于，它把测试从“代码片段”变成“行为描述”。阅读时不需要猜测每一步的意图，也不容易把准备逻辑、执行逻辑和断言逻辑混在一起。

## 3.3 三类最基本的测试路径
在实际业务中，一个方法通常至少需要考虑三类路径：

### 3.3.1 正常路径
正常路径验证“输入合法时，结果是否正确”。  
这是最基础的测试类型，主要用于确认主业务逻辑没有偏差。

### 3.3.2 异常路径
异常路径验证“输入非法或状态不满足时，是否按预期抛出异常或拒绝执行”。  
如果只测正常路径，很多真正危险的问题不会被及时发现。

### 3.3.3 边界路径
边界路径验证“在临界输入下，方法是否仍然稳定”。  
例如：
- 0
- 空字符串
- `null`
- 最大值/最小值
- 阈值附近的前后值

边界值往往是 bug 的高发区，因此不能只满足于“随便测几个正常值”。

## 3.4 常用断言
JUnit 的断言很多，但在日常业务开发中，最常见且最重要的通常是以下几类。

### 3.4.1 `assertEquals`
用于验证结果值是否符合预期。

```java
assertEquals(7, calculator.add(3, 4));
```

适合用于：
- 返回值验证
- 状态值验证
- 字符串、数字、对象字段的结果校验

### 3.4.2 `assertThrows`
用于验证某段代码是否抛出指定异常。

```java
assertThrows(IllegalArgumentException.class, () -> calculator.divide(10, 0));
```

适合用于：
- 非法参数
- 状态冲突
- 业务规则保护

在很多业务场景中，异常本身就是方法行为的一部分，因此异常路径并不是“额外补充”，而是正式业务逻辑的一部分。

### 3.4.3 `assertTimeout`
用于验证某段逻辑是否在限定时间内完成。

```java
assertTimeout(Duration.ofMillis(200), () -> {
    Thread.sleep(80);
    return "OK";
});
```

适合用于：
- 需要限制执行时长的本地逻辑
- 防止明显性能退化

它不等同于性能压测，但能作为一个轻量级性能保护手段。

## 3.5 案例解析：`CalculatorTest`
对应测试类：
- `src/test/java/com/company/training/core/CalculatorTest.java`

该测试类覆盖了两个典型场景：
- 加法正常路径
- 除法异常路径

还包含了一个边界值示例：
- `Integer.MAX_VALUE` 与 `0` 相加

这一点非常重要，因为它说明测试不是只验证“普通输入”，还应验证“临界输入下行为是否稳定”。

## 3.6 案例解析：`TimeoutAndDisabledTest`
对应测试类：
- `src/test/java/com/company/training/core/TimeoutAndDisabledTest.java`

该测试类展示了两个扩展点：

### 3.6.1 超时断言
`assertTimeout` 用于对执行时间加约束，适合本地可控逻辑。

### 3.6.2 临时禁用
`@Disabled` 表示某个测试当前不参与执行。  
这个注解应谨慎使用，通常只适用于以下情况：
- 外部依赖暂时不可用
- 某段示例仅用于演示语法
- 临时规避已知问题并等待修复

不建议把 `@Disabled` 当作规避失败测试的常态手段，否则测试体系会逐渐失去可信度。

## 3.7 测试命名建议
测试命名的目标不是简短，而是明确表达行为。  
推荐命名方式包括：
- `shouldAddTwoPositiveNumbers`
- `shouldThrowWhenDivideByZero`
- `shouldKeepMaxValueWhenAddZero`

这类命名有两个优点：
- 能直接说明测试目标。
- 报告失败时，能快速理解失败场景。

不推荐使用：
- `test1`
- `demo`
- `checkMethod`

因为这类名称不提供任何业务信息。

## 3.8 编写基础单测时的常见问题
### 3.8.1 测试里写了太多逻辑
如果测试本身包含复杂分支、循环或计算，很可能说明测试已经在“重写被测逻辑”。这会降低可信度。

### 3.8.2 一个测试验证过多内容
一个测试同时验证多个不相关行为，失败后定位成本会明显升高。

### 3.8.3 只测正常情况，不测异常和边界
这是最常见的覆盖不足问题。很多线上问题不是来自“主路径错误”，而是来自“边界路径无人验证”。

## 3.9 实践建议
- 先想清楚“这个方法最重要的行为是什么”，再写测试。
- 至少覆盖一条正常路径和一条异常或边界路径。
- 断言尽量靠近业务规则，不要只断言“代码跑完了”。
- 让测试代码像说明书一样容易读懂。

## 本章小结
高质量单元测试的第一步，是把业务规则准确、清晰地写成断言。AAA 结构提供了稳定的组织方式，`assertEquals`、`assertThrows` 和 `assertTimeout` 构成了最基础的表达工具。掌握这部分内容后，后续的参数化测试、Mock 和框架测试才有坚实基础。
