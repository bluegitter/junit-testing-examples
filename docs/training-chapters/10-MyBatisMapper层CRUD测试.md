# 第10章 MyBatis Mapper层CRUD测试

## 本章导读
在使用 MyBatis 的项目中，Mapper 层承担着 SQL 执行与结果映射的职责。很多团队在测试时容易忽略这一层，认为只要 Service 跑通即可。但从工程角度看，Mapper 层恰恰是最容易因 SQL 变更、字段映射变化或数据库脚本调整而出现回归问题的地方。本章聚焦 MyBatis Mapper 层的单独测试，说明为什么要测、测什么，以及如何利用 `@MybatisTest`、H2 内存数据库和 SQL 初始化脚本快速建立稳定的 CRUD 测试。

## 学习目标
- 理解 Mapper 层测试在数据访问质量保障中的作用。
- 掌握 `@MybatisTest` 的定位与适用范围。
- 掌握使用 H2 + `@Sql` 构建隔离数据库测试环境的方法。
- 能为 Mapper 层编写基础 CRUD 测试。

## 10.1 为什么要单独测试 Mapper 层
Service 层测试可以验证“业务流程是否走通”，但不能完全替代 Mapper 层测试。  
原因在于 Mapper 层关注的是更底层、更具体的数据访问行为，例如：
- SQL 是否正确执行
- 条件字段是否写对
- 主键是否正确回填
- 查询结果是否映射到正确字段
- 删除、更新是否只影响目标记录

如果这部分问题只依赖更上层测试来发现，定位通常会更慢，也更不容易判断问题到底出在 SQL、Mapper 还是业务编排。

因此，Mapper 层测试的价值可以概括为一句话：  
它是数据访问正确性的第一道防线。

## 10.2 Mapper 层测试要验证什么
对一个典型 CRUD Mapper，至少应验证以下几类行为：
- `insert`：是否插入成功，主键是否回填
- `selectById`：是否能查出正确数据
- `update`：是否只更新目标记录，字段值是否生效
- `delete`：删除后数据是否不可见
- `selectAll`：列表查询是否完整、排序是否符合预期

测试并不要求一次覆盖所有复杂 SQL 场景，但至少应覆盖最核心的访问路径。

## 10.3 为什么使用 `@MybatisTest`
在 Spring Boot 项目中，如果直接用 `@SpringBootTest` 测 Mapper，虽然也能工作，但成本偏高。  
`@MybatisTest` 的价值在于：
- 只加载 MyBatis 相关组件
- 自动配置嵌入式测试数据库
- 测试范围更小，执行更快
- 更符合“只验证数据访问层”的目标

可以把它理解为 MyBatis 层的测试切片。

这和前一章的 `@WebMvcTest` 类似，都是“为某一层提供最小可用测试上下文”。

## 10.4 H2 内存库的作用
如果 Mapper 测试依赖真实数据库，会带来一系列问题：
- 环境准备复杂
- 数据状态难控制
- 测试执行速度慢
- 团队成员本地环境不一致

H2 作为内存数据库，非常适合测试场景，因为它具备以下特征：
- 启动成本低
- 不依赖外部服务
- 每次测试可快速重建环境
- 易于与 Spring Boot 测试自动装配配合

这意味着：  
Mapper 层测试可以在本地、CI 和培训环境中保持一致性。

## 10.5 `@Sql` 为什么重要
即使使用 H2，如果每个测试共享同一份历史数据，也仍然会产生测试污染。  
为了解决这个问题，本项目使用：

```java
@Sql(scripts = "/sql/employee-schema.sql", executionPhase = Sql.ExecutionPhase.BEFORE_TEST_METHOD)
```

这表示：
- 每个测试方法执行前，都重新执行建表脚本
- 测试环境从一致状态开始
- 测试之间互不影响

对应脚本文件：
- `src/test/resources/sql/employee-schema.sql`

这是一种非常实用的数据库测试隔离方式，尤其适合培训项目和中小型测试样例。

## 10.6 案例结构说明
对应 Mapper：
- `src/main/java/com/company/training/spring/mapper/EmployeeMapper.java`

对应测试：
- `src/test/java/com/company/training/spring/mapper/EmployeeMapperMybatisTest.java`

对应建表脚本：
- `src/test/resources/sql/employee-schema.sql`

该示例采用注解式 SQL，包含五个核心方法：
- `insert`
- `updateById`
- `deleteById`
- `selectById`
- `selectAll`

这类设计非常适合培训，因为结构简单、职责清晰，便于把注意力集中在测试方法本身。

## 10.7 `insert` 测试关注点
在 `shouldInsertAndSelectById()` 中，测试重点包括：
- 插入返回影响行数是否为 1
- 自增主键是否成功回填到对象
- 再次按 ID 查询时是否能正确取回数据

这一点非常关键。  
如果只验证“插入成功”，而不验证“主键回填”和“查询结果”，那么很多映射问题仍然可能被遗漏。

## 10.8 `update` 测试关注点
在 `shouldUpdateById()` 中，流程通常分为：
1. 先插入一条原始数据
2. 修改对象字段
3. 执行更新
4. 再查询验证结果

更新测试应重点验证：
- 返回值是否为 1
- 数据库中的字段是否确实变更
- 更新是否作用于预期记录

如果只断言影响行数，而不做查询回查，那么“SQL 执行了，但更新结果不对”的问题仍然难以及时发现。

## 10.9 `delete` 测试关注点
在 `shouldDeleteById()` 中，应验证：
- 删除返回值是否正确
- 删除后再次查询是否为 `null`

删除测试的核心并不是“执行了 delete 语句”，而是“数据在后续读路径上确实不可见”。

## 10.10 `selectAll` 测试关注点
列表查询测试通常至少验证：
- 返回数量是否正确
- 返回顺序是否符合预期

在本项目中，SQL 使用了：

```java
@Select("SELECT id, name, email FROM employees ORDER BY id")
```

因此 `selectAll()` 的结果具备稳定顺序，这样在测试中就可以更明确地断言列表内容。

## 10.11 案例中的工程设计价值
当前示例虽然简单，但已经包含了 Mapper 层测试的几个重要工程思想：
- 每个测试方法独立验证一个 CRUD 行为
- 使用内存数据库避免外部依赖
- 使用 `@Sql` 保障数据隔离
- 通过“写后再读”的方式验证数据库最终状态

这些做法在真实项目中同样适用，只是 SQL 会更复杂、表结构更多。

## 10.12 常见误区
### 10.12.1 只测 Service，不测 Mapper
这会让 SQL 问题只能在更高层暴露，增加定位成本。

### 10.12.2 只断言影响行数，不回查数据
很多映射或字段问题无法通过行数发现。

### 10.12.3 测试共享数据状态
如果不做隔离，测试之间会互相污染，结果不稳定。

### 10.12.4 把 Mapper 测试写成完整业务流程测试
Mapper 测试应聚焦数据访问本身，不应承担过多业务层职责。

## 10.13 实践建议
- CRUD Mapper 至少为核心方法建立基础测试。
- 写入类操作建议搭配“写后再读”验证最终状态。
- 每个测试前重建表结构或清理数据，避免相互影响。
- 对复杂 SQL，优先测试结果正确性，而不是只看方法有没有执行。

## 本章小结
Mapper 层测试的核心目标，是确保 SQL 执行与结果映射行为准确无误。通过 `@MybatisTest`、H2 内存数据库和 `@Sql` 初始化脚本，可以低成本建立稳定、可重复的数据访问测试环境。掌握这一层之后，下一章将进一步进入 Service 与 Mapper 协作的集成测试场景。
