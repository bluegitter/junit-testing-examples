# 第05章 参数化测试基础

## 本章导读
在单元测试编写过程中，最常见的重复场景之一是：同一段业务规则需要针对多组输入反复验证。此时，如果继续用多个几乎相同的测试方法来覆盖，只会增加重复代码和维护负担。参数化测试的作用，就是把“固定测试逻辑”和“变化测试数据”分离开来，使测试更紧凑、更清晰，也更容易扩展。

## 学习目标
- 理解参数化测试的适用场景与边界。
- 掌握 `@ParameterizedTest` 的基本写法。
- 掌握 `@CsvSource`、`@ValueSource`、`@NullSource` 的常见使用方式。
- 能根据输入类型选择合适的数据源。

## 5.1 什么是参数化测试
参数化测试可以理解为：  
“用同一段测试逻辑，对多组不同输入分别执行验证。”

它的核心思想并不复杂：
- 测试逻辑保持不变。
- 输入数据成批提供。
- 每组数据独立执行一次测试。

从结构上看，参数化测试本质上是在复用测试方法，而不是复用断言结果。

## 5.2 为什么使用参数化测试
参数化测试最直接的收益有三个：

### 5.2.1 降低重复代码
如果多个测试方法只是输入不同、断言形式相同，那么这些方法通常可以合并为一个参数化测试。

### 5.2.2 提升覆盖率
由于增加一组测试数据的成本很低，因此更容易补充边界值、非法值和特殊值。

### 5.2.3 提高可维护性
当业务规则调整时，只需要修改一处测试逻辑，并根据需要新增或调整测试数据，而不必逐个维护大量结构相同的测试方法。

## 5.3 参数化测试适合什么场景
以下场景非常适合使用参数化测试：
- 相同规则需要对多组输入重复验证。
- 不同输入应得到不同但结构一致的输出。
- 需要覆盖大量边界值或非法值。
- 需要验证一组“输入 -> 期望结果”的映射关系。

例如：
- 字符串标准化
- 数值区间判断
- 会员等级对应不同折扣
- 参数合法性校验

## 5.4 什么情况下不适合参数化测试
参数化测试并不是“越多越好”。  
以下情况通常不适合直接参数化：
- 每组测试数据对应完全不同的测试意图。
- 每个场景都需要独立的前置准备逻辑。
- 测试体内部逻辑已经很复杂，再参数化会降低可读性。

判断标准很简单：  
如果多个测试之间真正相同的是“规则”，那么适合参数化；如果不同的是“场景本身”，则更适合拆成多个独立测试。

## 5.5 参数化测试的基本结构
JUnit 5 中最常见的写法是：

```java
@ParameterizedTest
@ValueSource(strings = {"A", "B", "C"})
void shouldHandleMultipleInputs(String input) {
    // act + assert
}
```

与普通 `@Test` 相比，参数化测试的关键区别在于：
- 使用 `@ParameterizedTest` 替代 `@Test`
- 使用参数源注解提供测试数据
- 方法参数由 JUnit 自动注入

## 5.6 常见参数源
### 5.6.1 `@CsvSource`
适用于“输入和期望值成对出现”的场景。

示例：
```java
@CsvSource({
    "'  alice  ', Alice",
    "'BOB', Bob"
})
```

适合用于：
- 输入与输出的映射关系验证
- 多参数业务规则验证

### 5.6.2 `@ValueSource`
适用于单参数批量验证。

示例：
```java
@ValueSource(strings = {"", "   "})
```

适合用于：
- 一组相似非法值
- 一组相似边界值

### 5.6.3 `@NullSource`
适用于空值输入验证。

示例：
```java
@NullSource
```

它通常用于验证：
- 是否抛出异常
- 是否返回默认值
- 是否触发空值保护逻辑

## 5.7 案例解析：`StringNormalizerParameterizedTest`
对应测试类：
- `src/test/java/com/company/training/core/StringNormalizerParameterizedTest.java`

这个测试类是参数化测试的基础示例，覆盖了三种典型输入类型：

### 5.7.1 正常输入
通过 `@CsvSource` 验证多个名称在去空格、大小写转换后的结果。

这类场景说明：  
同一规则可以对多组文本输入重复验证，而不必写多个独立测试方法。

### 5.7.2 空字符串输入
通过 `@ValueSource` 验证空串和空白串都返回 `UNKNOWN`。

这类场景说明：  
对一组同性质输入，参数化测试比逐个写测试方法更清晰。

### 5.7.3 `null` 输入
通过 `@NullSource` 验证 `null` 会触发异常。

这类场景说明：  
参数化测试不仅适用于正常输入，也非常适合异常和边界路径。

## 5.8 如何为参数化测试命名
如果参数化测试不设置名称模板，测试报告通常只会显示方法名，不利于快速定位具体失败数据。

推荐写法：

```java
@ParameterizedTest(name = "输入 {0}，输出应为 {1}")
```

这样在测试失败时，可以直接看到是哪组输入导致问题，从而提高排查效率。

## 5.9 编写参数化测试时的常见问题
### 5.9.1 把完全不同场景硬塞进一个参数化测试
这样会让测试方法内部出现过多分支判断，反而失去清晰性。

### 5.9.2 数据过多但没有分类
如果正常值、边界值、非法值全部混在一起，测试虽然能跑，但阅读成本很高。

### 5.9.3 忽略报告可读性
没有自定义名称模板时，失败报告很难快速定位是哪组数据出了问题。

## 5.10 实践建议
- 优先把“同一规则、多组输入”的场景改造成参数化测试。
- 将正常值、边界值、非法值分开组织，保持语义清晰。
- 对于字符串、数字、枚举等简单类型，优先使用内置参数源。
- 为参数化测试添加 `name` 模板，提升测试报告可读性。

## 本章小结
参数化测试的本质是“让规则复用、让数据变化”。它特别适合处理大量结构相似的输入验证场景，能够显著减少重复测试代码，并提升边界覆盖能力。掌握基础参数源之后，下一章将继续进入更复杂的数据组织方式。
