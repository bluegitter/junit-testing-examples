# 第13章 测试报告与覆盖率报告生成

## 本章导读
单元测试执行通过，只能说明“当前测试未失败”；但如果要从团队治理角度持续提升质量，还需要进一步回答两个问题：第一，哪些测试执行了、失败了、被跳过了；第二，哪些业务代码已经被测试覆盖，哪些仍处于盲区。测试报告和覆盖率报告分别回答这两个问题。本章聚焦报告生成与结果阅读，说明如何在当前项目中生成报告，以及这些报告在日常开发和 CI 流程中的实际价值。

## 学习目标
- 掌握生成测试报告与覆盖率报告的 Maven 命令。
- 理解 Surefire 报告与 JaCoCo 报告的区别。
- 能快速定位报告输出位置并读取关键指标。
- 理解如何将报告接入 CI 与质量平台。

## 13.1 为什么需要测试报告
如果只有命令行输出，开发者只能知道：
- 构建是否成功
- 有多少测试通过或失败

但无法高效回答以下问题：
- 哪个测试类失败了
- 哪些测试被跳过了
- 当前模块覆盖率是否明显不足
- 某次改动是否导致覆盖率下降

测试报告与覆盖率报告的价值在于：
- 把结果结构化、可视化
- 让测试质量能被追踪和治理
- 为代码评审和 CI 提供量化依据

## 13.2 当前项目使用的报告工具
本项目通过 Maven 插件生成两类报告：

### 13.2.1 Surefire 报告
由 `maven-surefire-plugin` 和 `maven-surefire-report-plugin` 负责。  
主要用于展示：
- 测试执行情况
- 失败/错误/跳过数量
- 每个测试类的结果明细

### 13.2.2 JaCoCo 报告
由 `jacoco-maven-plugin` 负责。  
主要用于展示：
- 行覆盖率
- 分支覆盖率
- 方法覆盖率
- 类覆盖率

两类报告关注点不同：
- Surefire 关注“测试执行结果”
- JaCoCo 关注“代码被测试覆盖到什么程度”

## 13.3 一键生成命令
在项目根目录执行：

```bash
mvn clean verify
```

该命令会完成：
- 清理旧构建产物
- 编译业务代码与测试代码
- 执行全部测试
- 生成单元测试报告
- 生成覆盖率报告

相比 `mvn test`，`mvn clean verify` 更适合阶段性验证、培训演示和 CI 流程，因为它会输出完整质量产物。

## 13.4 报告输出位置
执行完成后，常用报告可在以下目录查看：

- 单元测试原始报告（XML / TXT）
  - `target/surefire-reports/`
- 单元测试 HTML 报告
  - `target/site/surefire-report.html`
- 覆盖率 HTML 报告
  - `target/site/jacoco/index.html`
- 覆盖率 XML 报告
  - `target/site/jacoco/jacoco.xml`

如果只是本地查看可视化结果，通常优先打开两个 HTML 文件即可。

## 13.5 如何阅读单元测试报告
Surefire 报告通常重点展示以下指标：
- `Tests run`
  - 执行的测试总数
- `Failures`
  - 断言失败数量
- `Errors`
  - 运行异常数量
- `Skipped`
  - 被跳过的测试数量

实际工作中，最应优先关注的是：
- `Failures` 是否为 0
- `Errors` 是否为 0
- `Skipped` 是否合理

其中，`Skipped` 并不一定是坏事，但必须有明确原因。  
如果长期存在大量跳过测试，通常说明测试体系存在隐患。

## 13.6 如何阅读覆盖率报告
JaCoCo 报告常见维度包括：
- Instructions
  - 字节码指令覆盖率
- Branches
  - 分支覆盖率
- Lines
  - 代码行覆盖率
- Methods
  - 方法覆盖率
- Classes
  - 类覆盖率

从工程角度看，最有价值的不是追求“绝对高覆盖率”，而是结合业务判断：
- 核心业务类是否覆盖不足
- 异常分支和边界分支是否遗漏
- 某次改动是否导致关键类覆盖率下降

覆盖率的真正价值，不在于数字本身，而在于帮助识别测试盲区。

## 13.7 报告与覆盖率的关系
这两类报告常被混淆，但它们解决的问题不同：

- 测试报告回答：
  - 测试有没有执行
  - 哪些测试失败
  - 失败信息是什么

- 覆盖率报告回答：
  - 哪些代码被执行到了
  - 哪些分支还没有测试覆盖

因此，一个项目可能出现以下情况：
- 测试全通过，但覆盖率很低
- 覆盖率不低，但关键路径断言质量不足

这说明：
- 覆盖率不能替代测试质量
- 测试通过也不能替代覆盖分析

## 13.8 当前项目的报告解读示例
如果只是告诉“报告路径在哪里”，读者往往仍然不知道应该先看什么。  
因此，培训时最好结合当前项目结果做一次最小化解读。

可以按如下顺序阅读：
1. 先看 Surefire 报告首页，确认 `Failures` 和 `Errors` 是否为 0。
2. 再看 `Skipped` 是否存在，以及跳过是否具有明确原因。
3. 再看 JaCoCo 首页，优先关注核心包和刚修改过的类。
4. 最后进入单个类，查看是否仍有明显未覆盖的异常分支和边界分支。

以当前项目为例，若新增了 `OrderStatusServiceExternalCallTest` 这类异常覆盖较多的测试，重新生成报告后，应重点观察：
- 测试总数是否增加
- `OrderStatusService` 相关代码的分支覆盖率是否提升
- 是否覆盖到超时、连接异常、空响应、未知状态等分支

这类“带着问题读报告”的方式，比单纯记住报告路径更有工程价值。

## 13.9 当前项目中的应用价值
在本培训项目中，报告主要有两类用途：
- 培训演示
  - 让参与者看到测试执行结果和覆盖范围
- 工程基线
  - 为后续新增测试或重构提供对比依据

例如，在新增一个边界测试或异常测试后，可以重新执行 `mvn clean verify`，观察：
- 测试数量是否增加
- 覆盖率是否提升
- 哪些类或分支被新用例覆盖到了

## 13.10 与 CI 集成的建议
在团队工程中，推荐将以下动作纳入 CI：
1. 固定执行 `mvn clean verify`
2. 收集 `target/surefire-reports` 作为测试结果产物
3. 收集 `target/site/jacoco/jacoco.xml` 用于质量平台分析
4. 对关键模块设置最低覆盖率阈值

如果项目接入 SonarQube 等平台，通常重点使用的是：
- `jacoco.xml`
- Surefire XML 报告

这样可以让覆盖率趋势、失败测试趋势和质量门禁形成闭环。

## 13.11 常见误区
### 13.11.1 把覆盖率当成唯一目标
覆盖率高不等于测试质量高。  
如果断言无效或场景选择失真，覆盖率再高也不能提供足够保护。

### 13.11.2 只看总覆盖率，不看关键类
项目整体覆盖率可能不低，但核心类仍然缺少关键测试。

### 13.11.3 报告生成后无人阅读
如果报告只是生成但从未被关注，它就无法真正参与质量治理。

## 13.12 实践建议
- 开发阶段使用 `mvn test` 快速验证。
- 提交前或阶段性检查使用 `mvn clean verify` 生成完整报告。
- 优先关注失败测试和关键模块覆盖率变化。
- 用覆盖率报告发现盲区，但不要被单一数字绑架。

## 本章小结
测试报告回答“测试执行得怎么样”，覆盖率报告回答“代码覆盖得怎么样”。两者结合，才能让测试体系既有结果视角，也有盲区视角。在工程实践中，报告不应只是构建副产物，而应成为持续改进测试质量的重要依据。

## 思考与练习
- 执行一次 `mvn clean verify`，记录当前项目测试总数和覆盖率概况。
- 选择一个覆盖率较低的类，补充两个测试后重新生成报告，观察变化。
- 思考：如果一个类覆盖率很高，但线上仍频繁出问题，最可能是哪里出了问题？
