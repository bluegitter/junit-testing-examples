# 第02章 环境与项目结构

## 本章导读
单元测试能否稳定执行，首先取决于工程环境是否统一，其次取决于项目结构是否清晰。很多测试问题看起来出现在代码层，实际根源往往是依赖版本不一致、目录组织混乱、测试资源放置不规范，或者构建命令理解不准确。本章围绕“如何把测试工程搭起来并稳定运行”展开，为后续所有章节建立共同基础。

## 学习目标
- 理解本项目的运行环境与基础依赖要求。
- 熟悉 Maven 项目的标准目录结构与测试资源组织方式。
- 理解 `pom.xml` 中测试相关依赖和插件的职责。
- 掌握测试执行、报告生成和产物查看的基本命令。

## 2.1 基础环境要求
本项目基于以下环境构建：
- JDK：1.8.0
- 构建工具：Maven
- 项目组织方式：标准 Maven 项目结构

选择 JDK 1.8 的目的，是与大量企业现有 Java 项目保持一致，降低迁移门槛。在培训环境中，优先保证示例具有广泛兼容性，而不是追求最新语法特性。

## 2.2 为什么要使用标准 Maven 结构
使用标准项目结构的意义不只是“看起来统一”，而是为了让构建工具、测试框架、IDE 和 CI 平台能够按约定工作。

Maven 默认约定如下目录：

```text
src/main/java         业务源码
src/test/java         测试源码
src/test/resources    测试资源（SQL、配置、测试数据）
target/               构建输出目录
pom.xml               依赖、插件与构建入口
```

这种结构带来的直接收益包括：
- IDE 可以自动识别源码目录与测试目录。
- `mvn test` 可以自动编译并执行测试。
- 测试资源能自动加入测试运行时 classpath。
- CI 平台可以按默认规则收集测试结果与报告。

## 2.3 当前项目结构说明
本培训项目在标准结构基础上，进一步按业务场景划分了测试示例目录：
- `src/main/java/com/company/training/core`
  - 放置纯 Java 业务逻辑，适合讲解基础单元测试。
- `src/main/java/com/company/training/service`
  - 放置带依赖协作的服务类，适合讲解 Mockito。
- `src/main/java/com/company/training/spring`
  - 放置 Spring Boot 与 MyBatis 示例代码。
- `src/test/java/...`
  - 与主代码保持相同包结构，便于从测试反向定位被测对象。
- `src/test/resources/sql`
  - 放置数据库测试初始化脚本。

这种组织方式有一个重要优点：测试示例与被测代码在包结构上基本对齐，阅读时可以快速建立“测试类对应哪个业务类”的映射关系。

## 2.4 `pom.xml` 的作用
在 Maven 项目中，`pom.xml` 既是依赖清单，也是构建规则入口。本项目的测试能力主要由三部分组成：

### 2.4.1 测试依赖
- `junit-jupiter`
  - 提供 JUnit 5 的测试模型、断言与参数化能力。
- `mockito-junit-jupiter`
  - 提供 Mockito 与 JUnit 5 的集成扩展。
- `spring-boot-starter-test`
  - 提供 Spring Boot 测试基础设施，如 `@WebMvcTest`、`MockMvc`、`@SpringBootTest`。
- `mybatis-spring-boot-starter-test`
  - 提供 MyBatis 测试切片能力，如 `@MybatisTest`。
- `h2`
  - 提供测试用内存数据库，避免依赖外部数据库环境。

### 2.4.2 构建插件
- `maven-compiler-plugin`
  - 负责按 JDK 1.8 目标版本编译源码。
- `maven-surefire-plugin`
  - 负责执行单元测试。
- `maven-surefire-report-plugin`
  - 负责生成单元测试 HTML 报告。
- `jacoco-maven-plugin`
  - 负责统计测试覆盖率并生成覆盖率报告。

## 2.5 测试资源为什么放在 `src/test/resources`
测试往往不仅依赖代码，还依赖辅助资源，例如：
- SQL 建表脚本
- 测试专用配置文件
- 固定测试数据
- Mock 响应文件

这些资源放在 `src/test/resources` 的好处是：
- 不会混入正式产物。
- 测试运行时可以直接通过 classpath 访问。
- 能够与测试代码一起被版本管理。

本项目中的典型示例是：
- `src/test/resources/sql/employee-schema.sql`

该脚本用于 MyBatis 与 Spring Boot 数据库测试，在每次测试前重建表结构，保证测试环境一致。

## 2.6 常用命令与区别
在培训过程中，至少需要掌握以下两个命令：

### 2.6.1 执行测试
```bash
mvn test
```

该命令会完成：
- 编译主代码
- 编译测试代码
- 执行测试

适合本地快速验证。

### 2.6.2 执行测试并生成报告
```bash
mvn clean verify
```

该命令会在执行测试基础上继续生成：
- 单元测试报告
- 覆盖率报告

适合培训演示、CI 任务和阶段性质量检查。

## 2.7 构建产物位置
执行 Maven 后，常见输出会出现在 `target/` 目录下：
- `target/classes`
  - 编译后的业务类。
- `target/test-classes`
  - 编译后的测试类。
- `target/surefire-reports`
  - 测试原始报告（XML/TXT）。
- `target/site/surefire-report.html`
  - 单元测试 HTML 报告。
- `target/site/jacoco/index.html`
  - 覆盖率 HTML 报告。

理解这些目录的用途，有助于在测试失败或报告生成异常时快速定位问题。

## 2.8 常见环境问题
环境准备阶段最常见的问题通常包括：
- JDK 版本不一致，导致依赖或字节码不兼容。
- Maven 配置源异常，导致依赖无法下载。
- IDE 未正确识别测试目录，导致测试无法运行。
- 本地缓存污染，导致依赖解析或插件执行异常。

对于培训项目，建议先通过命令行执行 `mvn test` 验证环境，再使用 IDE 运行单个测试类。这样可以先确认问题属于“构建环境”还是“IDE 配置”。

## 本章小结
单元测试的稳定性并不只由测试代码决定，还由工程结构、依赖管理、资源组织和构建规则共同决定。掌握环境与项目结构后，后续章节中的测试写法、框架扩展和报告生成才有稳定基础。
